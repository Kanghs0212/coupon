# 🎟️ 대규모 트래픽 대응 티켓팅 예매 플랫폼 프로젝트 기획서

## 📝 프로젝트 개요 (Introduction)

본 프로젝트는 단기간에 엄청난 트래픽이 몰리는 대형 콘서트 및 뮤지컬의 '선착순 좌석 예매' 시스템을 모티브로 합니다. 단순한 수량 차감(Coupon) 방식을 넘어, [회원 - 콘서트 - 좌석 - 예매]로 이어지는 유기적인 도메인 흐름 속에서 데이터 정합성을 보장하고 고성능을 발휘하는 아키텍처를 구축합니다.

### ⚡ 기술적 기반 (Architectural Baseline)

본 프로젝트의 핵심 예매 메커니즘은 이전 단계에서 진행한 **"Redis 비동기 대기열 아키텍처"** 성능 실험 결과를 모태로 합니다.

* **이전 실험 지표:** 가상 사용자 3,000명 1초 동시 접속 환경에서 **평균 응답 속도 267ms, 초당 처리량(Throughput) 1,550.4/sec, 에러율 0.00%** 달성.
* [이전 쿠폰 시스템 최적화 과정 보기](https://github.com/Kanghs0212/coupon/blob/master/Redis_README.MD)
* **본 프로젝트의 확장 방향:** 락(Lock) 점유로 인한 동기식 DB 커넥션 병목 현상을 배제하기 위해, 사용자의 좌석 선택 요청을 Redis 메모리 계층(List/Set 구조)에서 초고속으로 먼저 선점(Queueing)한 뒤 백그라운드 Worker를 통해 고속으로 DB 갱신 및 영수증(Reservation)을 발행하는 비동기 이벤트를 그대로 계승합니다.

## 🏗 2. 데이터베이스 모델링 및 ER 다이어그램

복잡해진 비즈니스 요구사항에 맞춰 데이터의 중복을 제거하고 연관 관계를 명확히 정의한 논리적 구조입니다.

![diagram.png](coupon/src/main/resources/templates/diagram.png)

### 📊 연관 관계 핵심 설명

* **Member : Reservation (1 : N):** 한 회원은 여러 번 예매할 수 있지만, 특정 예매 내역은 단 한 명의 회원에게 속합니다.
* **Concert : Seat (1 : N):** 하나의 콘서트는 여러 개의 좌석을 가집니다.
* **Seat : Reservation (1 : 1):** 특정 콘서트의 특정 좌석(예: A열 1번)은 단 하나의 예매 내역(영수증)과 매핑됩니다. 이를 통해 동일 좌석에 대한 중복 예매를 원천 차단합니다.

## 🔍 3. 데이터베이스 정규화 분석 (Normalization Analysis)

본 시스템에 설계된 테이블 구조는 데이터 중복과 이상 현상(Anomaly)을 방지하기 위해 엄격한 제3정규형(3NF)을 만족하도록 설계되었습니다.

### ① 제1정규형 (1NF) 만족 여부

* **조건:** 모든 도메인이 원자 값(Atomic Value)으로만 구성되어야 합니다.
* **검증:** `Member`의 이름이나 `Seat`의 좌석명(`A열 1번`), `Reservation`의 예매 일시 등 모든 속성은 더 이상 분해할 수 없는 단일 원자 값을 가집니다. 따라서 **1NF를 완벽히 만족**합니다.

### ② 제2정규형 (2NF) 만족 여부

* **조건:** 1NF를 만족하고, 기본키가 복합키일 경우 기본키의 진부분집합이 결정자가 되는 '부분 함수 종속(Partial Functional Dependency)'이 없어야 합니다. (즉, 완전 함수 종속이어야 함)
* **검증:** 본 설계의 모든 테이블(`Member`, `Concert`, `Seat`, `Reservation`)은 복합키가 아닌 대리키(Surrogate Key)인 `id`(단일 컬럼)를 기본키로 사용합니다. 기본키가 단일 컬럼이므로 기본키의 일부에 종속되는 속성이 존재할 수 없습니다. 따라서 **2NF를 완벽히 만족**합니다.

### ③ 제3정규형 (3NF) 만족 여부

* **조건:** 2NF를 만족하고, 기본키가 아닌 일반 속성들 간에 '이행적 함수 종속(Transitive Functional Dependency)'이 없어야 합니다. (즉, X → Y, Y → Z 일 때 X → Z 가 성립하는 종속 관계 제거)
* **검증 (Seat 테이블 예시):**
    * id(기본키) → concert_id (좌석 ID는 콘서트를 결정함)
    * id(기본키) → seat_name (좌석 ID는 좌석 이름을 결정함)
    * 일반 속성인 seat_name → price 등의 관계가 성립하지 않습니다. (동일한 좌석 이름이라도 공연마다 가격이 다를 수 있으며, 가격은 오직 좌석 ID에 의해서만 결정됨)
    * 모든 비기본 속성들은 오직 기본키인 `id`에만 직접 종속될 뿐, 일반 속성을 거쳐 이행적으로 종속되지 않으므로 **3NF를 완벽히 만족**하며 삽입/삭제/수정 이상 현상이 발생하지 않습니다.

## 🚩 4. 핵심 아키텍처: 초고속 비동기 대기열 및 이중 방어 시스템

3,000명의 유저가 동시에 'A열 1번(단일 좌석)'을 클릭하는 피켓팅 상황을 완벽하게 통제하기 위해 다음과 같은 아키텍처를 설계했습니다.

### ✅ 1차 방어선: Redis 비동기 큐 (초고속 선점)
* 사용자가 좌석을 누르면, DB 트랜잭션을 열지 않고 Redis의 `setIfAbsent` (원자적 연산)를 활용하여 0.001초 만에 좌석의 소유권을 확정합니다.
* 승리한 단 1명의 요청만 Redis Queue에 적재하고 나머지 2,999명에게는 즉시 400 Bad Request(선점 실패)를 반환하여 WAS 및 DB 스레드 고갈을 원천 차단합니다.

### ✅ 2차 방어선: DB 유니크 제약 조건 (정합성 보장)
* 백그라운드 Worker 스레드가 Queue에서 요청을 꺼내 `Seat` 상태를 `RESERVED`로 변경하고 `Reservation(영수증)`을 발행합니다.
* 이때 JPA `@OneToOne` 매핑에 의한 유니크 제약조건(Unique Constraint)이 작동하여, 시스템 오류로 큐에 중복 요청이 들어가더라도 DB 단에서 무조건 에러(SQLState: 23505)를 뱉어내 갱신 손실(Lost Update)을 100% 방지합니다.

### 📊 부하 테스트 결과 (JMeter 증명)
* **테스트 시나리오:** 1초 만에 3,000명의 사용자가 1개의 좌석 동시 예매 요청
* **결과 지표:** * **에러율(방어율): 99.96%** (정확히 1건 성공, 2999건 튕겨냄)
    * **데이터 무결성:** `reservation` 테이블에 정확히 1개의 영수증 데이터만 적재됨.
    * **TPS:** 동기식 락 대비 수십 배 향상된 **1,550 TPS** 달성 (Tomcat 인프라 튜닝 병행)

## 🔒 5. 보안 및 실전 비즈니스 통합 (Phase 6)

실제 서비스와 동일한 사용자 경험과 보안을 제공하기 위해 Spring Security 인프라와 프론트엔드를 결합했습니다.

### 🛡️ Spring Security + JWT 인증
* 사용자가 회원가입(`BCrypt` 패스워드 암호화) 후 로그인하면 서버가 고유한 **JWT(JSON Web Token)**를 발급합니다.
* 예매 요청 시 `JwtAuthenticationFilter`가 토큰의 위변조를 검사하고, 토큰 내부에 숨겨진 `memberId` 클레임을 추출하여 안전하게 예매 대기열로 넘깁니다. (파라미터 조작 해킹 방지)

### 🎫 마이페이지 및 프론트엔드(React) 연동
* 백엔드의 `ReservationController`를 통해 로그인한 회원의 예매 내역(영수증)을 조회하는 API를 개발했습니다.
* React 기반의 SPA(Single Page Application) 클라이언트를 구축하여, **[회원가입 ➡️ JWT 로그인 ➡️ 티켓 예매 ➡️ 마이페이지 티켓 확인]** 으로 이어지는 실전 풀사이클을 완성했습니다.
