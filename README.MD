# 🚀 대규모 트래픽 처리를 위한 선착순 쿠폰 발급 시스템 (High Concurrency System)

단기간에 엄청난 트래픽이 집중되는 '선착순 이벤트(한정판 쿠폰 발급 등)' 환경을 모티브로 구축한 백엔드 시스템입니다.
본 프로젝트의 핵심 목표는 "대규모 동시 요청 상황에서 발생하는 데이터 정합성(Race Condition) 문제를 해결하고, 시스템의 성능 한계를 테스트 도구를 통해 수치화하여 최적의 아키텍처로 개선해 나가는 과정"을 증명하는 것입니다.

## 🛠 1. 개발 환경 및 기술 스택 (Tech Stack)

- **Language:** Java 17
- **Framework:** Spring Boot 3.5.14
- **Build Tool:** Gradle
- **Database:** * H2 Database (비즈니스 로직 및 동시성 단위 테스트용)
    - PostgreSQL 15 (실제 서비스 환경 및 부하 테스트용)
    - Redis (분산 락 제어 및 캐싱용)
- **ORM:** Spring Data JPA, Hibernate
- **Test & Performance:** JUnit 5, AssertJ, Apache JMeter
- **Infrastructure:** Docker & Docker Compose

## 🏗 2. 도메인 설계 및 테스트 주도 환경 구축

가장 핵심이 되는 `Coupon` 도메인에 비즈니스 로직을 응집시켜 객체지향적으로 설계했습니다.

- **Coupon Entity:** 쿠폰의 식별자(ID), 이름(Name), 남은 수량(Quantity)을 관리.
- **비즈니스 로직 캡슐화:** `decrease()` 메서드를 통해 남은 수량을 1씩 차감하며, 0 이하일 경우 `IllegalArgumentException`을 발생시켜 외부에서 임의로 수량을 조작하지 못하도록 방어 로직을 구성했습니다.

**[Phase 0.5: Sanity Test]**
본격적인 동시성 제어에 앞서, H2 인메모리 DB와 `@SpringBootTest`를 활용하여 스프링 컨텍스트 로드 및 기본 데이터 CRUD 로직이 완벽하게 동작함을 검증했습니다.

## 🚩 3. 핵심 트러블슈팅 및 성능 최적화 과정

### ❌ Phase 1: 다중 트래픽 상황 시뮬레이션 및 문제 직면 (Race Condition)

- **목표:** 100명의 사용자가 동시에 쿠폰 발급을 요청하는 극단적인 환경 시뮬레이션.
- **구현:** `ExecutorService`를 통해 32개의 스레드 풀을 생성하고, `CountDownLatch`를 사용하여 100개의 비동기 요청이 동시에 `decrease()` 비즈니스 로직에 접근하도록 강제하는 테스트 코드 작성.
- **결과 (의도된 실패):** 100개의 쿠폰이 0개가 되어야 하지만, 실제 실행 결과 83개가 남아있는 심각한 데이터 불일치 발생.
- **원인 분석 (Root Cause):** 수많은 스레드가 DB에서 동일한 쿠폰 수량(예: 100개)을 동시에 읽어오고, 각자의 메모리에서 1을 차감한 값(99개)을 동시에 덮어쓰면서 다른 스레드의 갱신 내역이 날아가는 **갱신 손실(Lost Update)** 현상임을 확인했습니다.

### ✅ Phase 2: RDBMS 비관적 락(Pessimistic Lock)을 통한 정합성 보장

- **목표:** DB가 제공하는 Lock을 활용하여 갱신 손실 문제 원천 차단.
- **구현:** Spring Data JPA의 `@Lock(LockModeType.PESSIMISTIC_WRITE)` 도입. 쿼리 레벨에서 `SELECT ... FOR UPDATE`를 발생시켜, 특정 스레드가 트랜잭션을 마칠 때까지 다른 스레드의 Read/Write 접근을 차단했습니다.
- **결과:** 동일한 100명 동시 요청 단위 테스트 결과, 남은 수량이 정확히 '0'으로 떨어지며 데이터 정합성을 100% 보장하는 데 성공했습니다.

### 💣 Phase 2-1: 부하 테스트를 통한 비관적 락의 치명적 한계 도출

데이터의 정합성은 확보했으나, 락(Lock) 점유로 인한 병목이 실제 시스템에 미치는 영향을 검증하기 위해 Docker + PostgreSQL 환경을 구축하고 Apache JMeter로 부하 테스트를 수행했습니다.

- **테스트 시나리오:** 1만 장의 쿠폰이 있는 DB에 가상 사용자 3,000명이 1초 만에 동시 접속하여 API(`POST /coupons/1/decrease`) 호출.
- **테스트 결과 (JMeter 지표 분석):**
    - **데이터 무결성:** 3,000개의 요청 처리 후 수량이 정확히 7,000개로 남음 (정합성 완벽 유지).
    - **초당 처리량 (Throughput):** `27.9 / sec` (TPS: 약 28)
    - **평균 응답 시간 (Average):** `4,429 ms` (최대 13초 지연)
- **인사이트 및 한계점:** 하나의 트랜잭션이 row 레벨의 락을 쥐고 있는 동안 나머지 수천 개의 요청이 무한 대기 상태에 빠집니다. 결국 DB 커넥션 풀이 고갈되고 애플리케이션의 응답 속도가 4초 이상으로 치솟아, 실제 대규모 서비스에서는 사용할 수 없는 아키텍처임을 수치로 증명했습니다.

## 🎯 4. Next Step: Redis 분산 락(Distributed Lock) 아키텍처 도입

비관적 락이 야기한 심각한 성능 저하(TPS 28) 현상을 극복하기 위해, 디스크 I/O에 의존하는 RDBMS 락 대신 초고속 인메모리(In-Memory) 저장소인 Redis를 활용한 분산 락 도입을 결정했습니다.

- **구현 계획:** Redisson 라이브러리의 Pub/Sub 구조를 활용하여 락 획득을 위한 스핀 락(Spin Lock) 부하를 최소화.
- **목표:** 데이터 정합성을 100% 유지하면서도 TPS를 획기적으로 개선하고 응답 지연을 밀리초(ms) 단위로 단축하여 결과를 JMeter 지표로 재비교할 예정입니다.

### 📊 4.1 [Deep Dive] Redis 분산 락 도입 결과 및 병목 현상 분석

**1. 개요 (Background)**
선착순 쿠폰 발급 시스템에서 동시성 문제를 해결하기 위해 RDBMS의 비관적 락(Pessimistic Lock)을 적용했으나, 심각한 성능 저하(TPS 약 28)가 발생했습니다. DB의 디스크 I/O 부하를 줄이고 성능을 극대화하기 위해 인메모리(In-Memory) 기반의 Redis 분산 락(Redisson) 아키텍처를 도입하여 부하 테스트를 진행했습니다.

**2. 테스트 환경 및 시나리오**

- 테스트 도구: Apache JMeter
- 시나리오: 1만 장의 잔여 쿠폰 상태에서, 가상 사용자 3,000명이 1초 내에 동시 발급 요청 (`POST /coupons/1/decrease`)
- 방어 로직 (Facade): 락 획득 대기 시간 10초 설정 (10초 초과 시 예외 처리하여 무한 대기 방지)

**3. 테스트 결과 지표 (JMeter)**
예상과 달리, Redis 분산 락을 도입했음에도 전체적인 성능 지표가 오히려 하락하는 기현상을 확인했습니다.

- 초당 처리량 (Throughput): `27.9 / sec` ➡️ **`8.2 / sec` (성능 저하)**
- 평균 응답 시간 (Average): `4,429 ms` ➡️ **`5,406 ms` (지연 증가)**
- 에러율 (Error Rate): `0.00%` ➡️ **`34.00%` (대량의 Connection Timeout 발생)**
- 데이터 정합성: 쿠폰 2,940장 정상 발급 (DB 잔여 7,060장). 👉 *나머지 60건은 10초 대기 시간 초과로 정상적인 락 획득 포기(방어 로직 성공).*

**🔍 4. 원인 분석 (Root Cause Analysis): 왜 성능이 떨어졌는가?**
Redis 자체는 초당 수십만 건을 처리할 수 있을 만큼 빠릅니다. 성능 저하의 진짜 원인은 Redis가 아니라 애플리케이션(Tomcat)과 DB(PostgreSQL) 사이의 스레드 및 커넥션 자원 고갈에 있었습니다.

- **① Tomcat Thread Pool 고갈 (Error 34%의 원인):**
  Spring Boot(내장 Tomcat)가 동시에 처리할 수 있는 기본 스레드 수(Max Threads)는 200개입니다. 1초 만에 3,000개의 요청이 몰려오면, 200개의 스레드가 락을 획득하거나 락을 기다리며 대기 상태(Block)에 빠집니다. 나머지 2,800개의 요청은 스레드를 할당받지 못해 큐에서 대기하다가 결국 Connection Refused / Timeout 에러를 뱉고 튕겨 나가게 됩니다.
- **② HikariCP Connection Pool 병목 (TPS 저하의 원인):**
  Redis에서 락을 매우 빠르게 획득했다 하더라도, 실제 쿠폰 수량을 차감하려면 결국 DB에 `UPDATE` 쿼리를 실행해야 합니다. 이때 Spring Boot의 기본 DB 커넥션 풀(HikariCP) 사이즈는 10개입니다. 즉, 락을 얻은 수백 개의 스레드라 할지라도 DB 접근을 위해 또다시 10개의 커넥션을 두고 이중 대기(Secondary Bottleneck)를 하게 됩니다. 자물쇠만 빨리 풀었을 뿐, 최종 목적지(DB)로 들어가는 문은 여전히 좁았기 때문에 TPS가 오를 수 없었습니다.

## 🎯 5. 최종 아키텍처 개선 방향: 동기식 락 ➡️ 비동기 대기열(Queue)

이러한 테스트 결과를 통해, 트래픽이 폭증하는 환경에서는 "어떤 방식의 Lock이든 동기식(Synchronous)으로 DB를 즉시 업데이트하는 구조 자체에 한계가 있음"을 증명했습니다.
이를 근본적으로 해결하기 위해 락(Lock)에 의존하는 방식에서 벗어나, 메시지 큐(Redis List 또는 Kafka)를 활용한 비동기(Asynchronous) 처리 아키텍처로 시스템을 전면 개편했습니다.

**[개선된 아키텍처 흐름]**

1. **API 요청 (Fast):** 사용자의 요청이 오면 DB를 거치지 않고 메모리(Redis 큐)에 대기표만 발급한 뒤 즉시 200 OK 응답 (Tomcat 스레드 점유 최소화).
2. **백그라운드 처리 (Async):** 별도의 Worker 스레드가 큐에서 요청을 꺼내어, DB 커넥션 풀이 감당할 수 있는 속도(예: 한 번에 10개씩)로 DB에 Bulk Update 수행.

## 🚀 6. 극한의 인프라 튜닝 및 부하 테스트 완료 (Error 0% 달성)

비동기 대기열 구조 적용으로 응답 속도와 TPS는 극적으로 향상되었으나, 1초에 3,000명의 동시 접속 시 OS 레벨의 네트워크 소켓(TCP Backlog) 한계로 인해 여전히 18%의 `Connection Refused` 에러가 발생하는 현상을 마주했습니다. "단 한 명의 사용자도 튕기지 않는 완벽한 시스템"을 목표로 최후의 인프라 튜닝을 단행했습니다.

**[최종 튜닝 내역]**

- **Tomcat 인프라 확장:** `application.yml`에서 `max-threads: 1000`, `accept-count: 10000`, `max-connections: 10000`으로 상향 조정하여 서버의 출입구 및 대기열을 극한으로 확장.
- **클라이언트(JMeter) 병목 해결:** 초당 3,000 소켓을 뚫어내는 클라이언트 툴 자체의 한계를 우회하기 위해 HTTP 통신 구현체를 `Java Native`로 변경.
